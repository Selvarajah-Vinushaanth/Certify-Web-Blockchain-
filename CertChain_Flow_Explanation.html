2. Issuing a Certificate — Step by Step

Step 1 — Fill the Form (Next.js Frontend) 
You enter:
  - Student Name
  - Student Email
  - Course / Degree
  - Student Wallet Address  ← recipient identity (public address)
  - Upload PDF file

Click: "Issue Certificate"
↓
Step 2 — Next.js API processes the request 
File: src/app/api/certificates/route.ts

  1. Reads PDF as raw bytes
  2. Hashes the PDF:
       certHash = keccak256(pdf_bytes) = 0x3f7a9b...
       (unique 32-byte fingerprint — changes if PDF is edited even 1 bit)
  3. Sends PDF to Pinata API
↓
Step 3 — Pinata (IPFS) stores the PDF 
PDF uploaded to Pinata's servers (permanent IPFS storage)

Pinata returns:
  CID = "QmXyz123abc..."

CID = Content Identifier — permanent address to the file
  - Same file always gives same CID
  - File can never be deleted from IPFS
  - Accessible at: gateway.pinata.cloud/ipfs/QmXyz123abc...

NOTE: CID is NOT a cache — it is a PERMANENT link.
↓
Step 4 — PostgreSQL saves a pending record 
Database saves:
  {
    certId:       "pending-1234-xyz"  ← temporary placeholder
    certHash:     "0x3f7a9b..."
    ipfsCid:      "QmXyz123..."
    studentName:  "Vinushaanth"
    courseName:   "CSE"
    recipientAddr:"0x70997..."        ← student wallet
    issuerAddr:   "0xf39Fd6..."       ← your wallet
    status:       ACTIVE
  }

Why save now?
  → IPFS upload already happened (can't undo)
  → Need a record even if blockchain step fails
  → certId will be updated after blockchain confirms
↓
Step 5 — MetaMask signs the transaction 
Frontend calls the Smart Contract function via MetaMask:
  contract.issueCertificate(certHash, CID, studentWallet, name, course)

MetaMask pops up showing:
  - Function being called
  - Network: Hardhat Local  ← MUST be this (not Ethereum Mainnet)
  - Gas fee (small amount of test ETH)

You click CONFIRM →
  MetaMask uses your PRIVATE KEY to sign:
    signature = ECDSA.sign(transactionHash, privateKey)

Private key NEVER leaves MetaMask — signing happens internally.
The signature proves: "I, the owner of this wallet, authorized this."
↓
Step 6 — Hardhat Blockchain processes the transaction 
Hardhat (localhost:8545) receives the signed transaction:

  1. VERIFIES SIGNATURE:
       recoveredAddress = ECDSA.recover(txHash, signature)
       → Proves transaction was signed by 0xf39Fd6...

  2. CHECKS AUTHORIZATION:
       require(authorisedIssuers[msg.sender] == true)
       → Only registered issuers can issue certificates

  3. GENERATES CERTIFICATE ID:
       certId = keccak256(certHash + issuerAddress + timestamp)
       = 0x6748e1466c...

  4. STORES PERMANENTLY ON BLOCKCHAIN:
       certificates[certId] = {
         certHash, ipfsCid, issuer, recipient,
         studentName, courseName, issuedAt, valid: true
       }
       → Cannot be edited or deleted EVER

  5. RETURNS: certId + transactionHash to frontend
↓
Step 7 — PostgreSQL updated with real Certificate ID 
Frontend receives certId from blockchain

Calls PATCH /api/certificates/{id}:
  certId:  "pending-1234-xyz" → "0x6748e1466c..."
  txHash:  "0xb204bc3d..."

Database now has the complete, real record.
Dashboard shows the real Certificate ID.

DONE ✓

5. Verifying a Certificate

Method 1 — By Certificate ID

Go to localhost:3000/verify
Paste certId: 0x6748e1466c...
        ↓
Frontend calls: contract.verifyCertificate(certId)
        ↓
Hardhat returns: valid, studentName, courseName, issuer, issuedAt, ipfsCid
        ↓
Shows: VALID ✓ (or REVOKED ✗)

Method 2 — By uploading PDF

Upload the PDF file
        ↓
Browser hashes it: fileHash = keccak256(pdf_bytes)
        ↓
Calls: contract.verifyCertificateByHash(fileHash)
        ↓
Blockchain searches for matching hash

FOUND    → Certificate is REAL ✓  (PDF not tampered)
NOT FOUND → Certificate is FAKE ✗  (PDF was edited)

7. Simple Analogy

Think of it like a notary stamp on a document:

  • Private key = the notary's unique stamp (only they have it)
  • Public key = the notary's name (everyone can verify who stamped it)
  • Certificate ID = the unique serial number on the stamped document
  • Blockchain = the government registry that records the stamp forever
  • Pinata/IPFS = the archive room where the actual document is stored
